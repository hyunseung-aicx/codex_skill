# 처음 접하는 분께 — 큰 그림부터 잡기

> 이 문서는 **클로드 코드(Claude Code) 자체를 처음 써본 분**, 또는 **써봤지만 "하네스 엔지니어링", "훅", "에이전트" 같은 단어가 낯선 분**을 위한 입문서입니다.
> 비개발자도 읽을 수 있도록 비유 위주로 풀어 썼습니다. 코드 한 줄 없이 끝까지 읽으실 수 있습니다.

---

## 🚗 큰 그림 한 줄 비유

**클로드 코드는 "운전자"이고, 이 레포는 그 운전자가 타는 "자동차의 안전·편의 장치 모음"입니다.**

운전자(클로드 코드)는 똑똑하지만 혼자만의 판단으로 운전하면 사고가 납니다. 그래서 차에는:

| 자동차 장치 | 이 레포의 대응물 | 역할 |
|------------|----------------|------|
| ABS, 에어백, 차선이탈 경보 | **훅(hooks)** | 위험 순간 자동 작동, 사고 예방 |
| 정비공, 보험사, 변호사 | **에이전트(agents)** | 전문 영역을 위임받는 별도 인격 |
| 차량 정비 기록부 | **메모리(memory-schema)** | "이 차는 작년에 사고 났음"처럼 사실을 기록 |
| 운전 매뉴얼 | **스킬(skills)** | 자주 쓰는 절차의 정형화 |
| 변환기, 정비 도구 | **스크립트(scripts)** | 일회성 작업용 도우미 |
| 좌석 조정, 거울 각도 | **설정(settings)** | 운전자가 차를 자기 몸에 맞추는 부분 |

이 비유를 머릿속에 두고 아래로 내려가시면 됩니다.

## 📂 폴더별 — 어디서 무슨 일이 일어나는가

```
claude_skill/
│

├── settings/       ⚙️ 자동차에 안전장치를 연결하는 설계도
├── scripts/        🔧 일회성 도구
└── docs/           📚 사용설명서 (README, SETUP_GUIDE 등)
```

### 🛡️ hooks/ — "위험 순간에 자동으로 작동하는 안전장치"

#### 비유
운전 중 옆 차선에 차가 너무 가깝게 붙으면 **차선이탈 경보**가 자동으로 울립니다. 운전자가 매번 "옆 차 거리 봐야지!" 신경 쓰지 않아도 되도록.

훅(hook)도 똑같습니다. 클로드 코드가 도구를 사용하거나 메시지를 받을 때 **자동으로 끼어들어** 검사·기록·차단·추천 등을 수행합니다.

#### 왜 필요한가
사고 사례 두 개로 설명드리겠습니다:

- **$47,000 / 11일 사고 (2026.03)**: 어떤 분이 클로드 에이전트를 자율 모드로 돌렸는데, 에이전트 A와 에이전트 B가 같은 코드를 서로 되돌리는 무한 루프에 빠졌습니다. 11일 동안 아무도 모르고 있다가 $47,000(약 6,500만 원) 청구서가 날아왔습니다.
- **$4,200 / 63시간 사고 (2026.02)**: API rate limit이 걸렸는데 에이전트가 무한 재시도를 했습니다. 63시간 만에 $4,200(약 580만 원).

이 두 사고 모두 **훅이 있었다면 첫날 첫 단계에서 막혔습니다**. 그래서 이 레포에 8개 훅을 둡니다.

#### 이 레포의 훅 8개 — 한 줄 설명

| 훅 | 역할 | 일상 비유 |
|----|------|----------|
| `budget-gate.sh` | 누적 비용이 $5(세션) / $20(하루)를 넘으면 자동 차단 | 가계부에 "이번 달 외식 10만 원 초과 시 알림" |
| `prompt-cache-monitor.sh` | 클로드 API의 캐시 사용률 측정 | 자동차 연비 모니터링 (낮으면 점검 권유) |
| `llm-judge.sh` | 작업 끝날 때마다 "AI가 시킨 일을 제대로 했는가" 다른 AI에게 검토받음 | 시험 보고 채점받기 — 자기 채점 금지 |
| `tool-hash-dedup.sh` | 같은 동작을 5번 이상 반복하면 "뭐가 잘못된 것 같으니 멈춰" 신호 | 자동차가 같은 자리에서 5번 후진하면 GPS가 "다른 길로?" 제안 |
| `model-router.sh` | 작업 내용 보고 "이건 작은 일이니 Haiku(저렴)", "이건 큰 일이니 Opus(고성능)" 추천 | 일의 크기에 따라 다른 사람에게 맡기는 매니저 |
| `tool-selector.sh` | 작업에 필요한 외부 도구만 활성화 (예: Slack 안 쓰면 끔) | 외출할 때 필요한 가방만 챙기기 |
| `otel-trace-exporter.sh` | 모든 활동을 표준 포맷(OpenTelemetry)으로 기록 | 운행일지를 표준 양식으로 작성 (나중에 분석 쉬움) |
| `bench-runner.sh` | 매일 밤 자동으로 미니 테스트 20개 돌려서 셋업이 잘 작동하는지 확인 | 매월 자동차 정기 점검 |

#### 쉬운 예제

여러분이 클로드 코드한테 "이 코드 리팩토링해줘"라고 부탁했다고 상상해보세요.

```
1. 메시지 입력 → model-router 가 "이건 리팩토링이니 Opus 모델 쓰는 게 낫겠다" 추천
2. 클로드가 작업 시작 → 매 도구 사용마다 budget-gate 가 비용 계산
3. 클로드가 같은 파일을 5번째 수정하려고 함 → tool-hash-dedup 이 경고
4. 작업 종료 → llm-judge 가 "변경사항이 사용자 요청에 충실한가?" 평가
5. 백그라운드 → otel-trace-exporter 가 모든 활동을 기록
```

여러분은 그냥 "리팩토링해줘"만 했는데, **5개의 안전장치가 자동으로 돌면서 사고 방지·비용 통제·품질 검증**을 동시에 수행합니다.

### 🤖 agents/ — "전문가 인격을 분리해서 부르기"

#### 비유
회사에서 법적 분쟁이 나면 **변호사**를 부르고, 세금 문제는 **회계사**를 부릅니다. 사장이 직접 모든 분야를 다루지 않습니다. 각 전문가는 자기 영역에 집중해서 더 깊은 답을 줍니다.

에이전트(agent)도 똑같습니다. 클로드 코드 메인 세션은 매니저 역할을 하고, 전문 작업은 **에이전트라는 별도 인격**에게 위임합니다.

#### 왜 필요한가
1. **컨텍스트 분리**: 메인 세션이 너무 많은 일을 기억하면 헷갈립니다. 에이전트는 자기 일만 보고 결과만 보고.
2. **전문성**: 보안 검토 전문 에이전트는 OWASP Top 10을 머릿속에 항상 두고 있고, Python 리뷰어는 type hint 규칙을 항상 적용합니다.
3. **병렬 처리**: 보안·성능·접근성을 동시에 검토할 수 있습니다.

#### 기존 24개 에이전트
프로젝트 종류별로 이미 24개 에이전트가 있습니다:
- `planner` (기획), `code-reviewer` (코드 리뷰), `security-reviewer` (보안)
- `python-reviewer`, `go-reviewer`, `react-agent` (언어/프레임워크별)
- 등등 ...

#### 이 레포가 추가하는 1개 — `judge-agent`

**왜 필요한가**: 위 24개 에이전트는 모두 "**일을 하는**" 에이전트입니다. 그런데 그 결과를 **종합적으로 판정**하는 에이전트가 없었습니다. 

예를 들어 코드 리뷰어가 "괜찮음"이라고 했고, 보안 검토자가 "통과"라고 했고, 테스트가 통과했는데도 — 정말로 **사용자가 원한 걸 만들었는가**는 별개의 질문입니다.

`judge-agent`는 모든 증거를 모아서 5가지 기준으로 1~5점씩 채점합니다:

| 기준 | 무엇을 보는가 |
|------|--------------|
| 충실도 (Faithfulness) | 사용자가 요청한 걸 정말 만들었는가, 아니면 옆길로 샜는가 |
| 테스트 무결성 (Test integrity) | 테스트를 통과시키려고 테스트를 약화시키지 않았는가 |
| 보안 (Security) | 하드코딩된 비밀번호, SQL 인젝션 위험 등 |
| 코드 품질 (Code quality) | 함수 크기, 중복, 디버그 로그 잔존 등 |
| 되돌릴 수 있는가 (Reversibility) | 잘못됐을 때 안전하게 롤백 가능한가 |

총점 19점 이상이면 PASS, 12점 이하면 BLOCK, 그 사이는 "주의해서 통과".

#### 쉬운 예제

```
사용자: "로그인 기능 만들어줘"
↓
메인 클로드: 코드 작성
↓
code-reviewer 에이전트: "함수가 너무 크네요. 분리 권장" (점수: 3/5)
security-reviewer 에이전트: "비밀번호 평문 저장 위험!" (점수: 2/5)
tdd-guide 에이전트: "테스트 없음" (점수: 1/5)
↓
judge-agent: 
    "충실도 4 + 테스트무결성 1 + 보안 2 + 품질 3 + 가역성 4 = 14점
     테스트무결성 1점이라서 BLOCK"
↓
메인 클로드: 테스트 먼저 작성 후 재시도
```

### 🧠 memory-schema/ — "시간이 흐르면 사실이 바뀐다"

#### 비유
자동차 정비 기록부에 "2024년 브레이크 교체"라고 적혀 있어도, **5년 뒤에는 다시 교체해야 합니다**. 기록은 **언제 유효했는지**를 알아야 의미가 있습니다.

#### 왜 필요한가
클로드 코드의 기존 메모리는 **append-only**(추가만 가능)였습니다. 그러다 보니 6개월 후에는:
- "이 프로젝트는 Python 3.9 사용" (낡은 기록)
- "이 프로젝트는 Python 3.13 사용" (새 기록)

두 가지가 공존하고, 클로드가 어느 게 진짜인지 모릅니다.

#### 해결책: Zep 패턴 도입
Zep은 메모리에 **시간 차원**을 추가한 시스템입니다. 학술 측정(LongMemEval)에서 **+14.8점** 개선을 보였습니다. 핵심 아이디어 4가지:

| 필드 | 의미 | 예 |
|------|------|----|
| `valid_from` | "이 사실은 언제부터 유효한가" | 2026-05-14 |
| `valid_until` | "언제까지 유효한가" (null이면 현재) | 2027-01-01 |
| `supersedes` | "이전 어떤 기록을 덮어쓰는가" | `python-3-9-version` |
| `confidence` | "얼마나 확신하는가" (0~1) | 0.95 |

#### 쉬운 예제

```yaml
# 2026-05-14에 작성된 메모리
---
name: project-python-version
description: 이 프로젝트는 Python 3.13 사용
metadata:
  type: project
  valid_from: "2026-05-14"
  valid_until: null
  supersedes: ["project-python-3-9-version"]   # ← 이전 메모리 무효화
  confidence: 0.95
---
```

이제 클로드가 "Python 버전이 뭐였더라?" 물어보면:
- 옛 메모리 → "이건 supersedes 됐네, 무시"
- 새 메모리 → "3.13. 확신도 95%"

라고 답할 수 있습니다.

### ⚙️ settings/ — "안전장치를 자동차에 연결하는 설계도"

#### 비유
ABS가 차에 달려 있어도 **전선이 연결돼야** 작동합니다. settings 폴더가 그 전선 역할입니다.

#### 왜 필요한가
훅(`hooks/*.sh`)을 만들기만 해서는 작동하지 않습니다. 클로드 코드에게 **"어떤 순간에 어느 훅을 부를지"** 알려줘야 합니다.

settings.json 파일이 그 역할을 합니다:
- `SessionStart` (세션 시작) 시 → 어떤 훅?
- `PreToolUse` (도구 사용 직전) 시 → 어떤 훅?
- `Stop` (작업 종료) 시 → 어떤 훅?

#### 이 레포의 settings.example.json
**바로 쓸 수 있는 예시 파일**입니다. 본인의 `~/.claude/settings.json`에 머지(병합)하면 됩니다. 자세한 머지 방법은 [`SETUP_GUIDE_macOS.md`](./SETUP_GUIDE_macOS.md) 6절을 참조하시면 됩니다.

#### 쉬운 예제

```json
{
  "hooks": {
    "Stop": [{
      "hooks": [{
        "command": "bash ${CLAUDE_SKILL_DIR}/hooks/llm-judge.sh"
      }]
    }]
  }
}
```

해석: "작업이 끝날 때(Stop) `llm-judge.sh` 훅을 실행하라"

### 🔧 scripts/ — "일회성 도우미"

#### 비유
자동차에 **타이어 펌프**가 항상 달려 있을 필요는 없습니다. 1년에 몇 번만 쓰니까 트렁크에 보관합니다. 스크립트도 마찬가지로 가끔 쓰는 도구입니다.

#### 왜 필요한가
훅은 매번 자동으로 돌아가야 하지만, 스크립트는 **사용자가 직접 부를 때만** 실행됩니다. 마이그레이션, 일회성 변환, 셋업 같은 작업에 적합합니다.

#### 이 레포의 스크립트
- `migrate-to-otel.sh`: 기존에 쌓인 추적 로그를 OpenTelemetry 표준 포맷으로 일괄 변환

#### 쉬운 예제

```bash
# 작년 1년치 추적 로그를 한 번에 변환
bash scripts/migrate-to-otel.sh ~/.claude/traces/2025.jsonl > /tmp/otel-2025.jsonl

# 이제 Datadog/Langfuse 같은 외부 도구에 import 가능
```

### 예제 1️⃣ — 평범한 프롬프트 (대부분의 경우)

여러분이 그냥 "버그 고쳐줘" 라고 입력하면 — **아무 추가 메시지도 안 보입니다**.

```
$ claude

> 버그 고쳐줘

[클로드가 평소처럼 작업 시작...]
```

이게 정상입니다. 후크는 "위험 / 비효율 / 사고 가능성"이 감지될 때만 끼어듭니다.

---

### 예제 2️⃣ — 짧은 작업 (Haiku 권장)

```
$ claude

> 오타 고쳐줘

🧠 모델 라우팅 v2 추천:
  ▸ 모델: haiku (low)
  ▸ Complexity: 0.0/10
  ▸ 예상 비용: $0.04 (입력 40,001토큰, 출력 500토큰)
  ▸ 이유: 단순 키워드 1개

[클로드가 작업 시작 — 사용자가 원하면 /model 로 haiku 전환]
```

해석:
- 짧고 단순한 작업이라 **Haiku로 충분하다고 추천**합니다.
- 사용자가 결정 — 그대로 Sonnet 쓰셔도 되고, `/model haiku` 명령어로 전환하셔도 됩니다.
- 한 달간 매일 이런 작업 100번 하면 Haiku로 전환 시 **약 $90 절감**.

### 예제 3️⃣ — 복잡한 작업 (Opus + subtask 분해)

```
$ claude

> 인증 시스템 전체를 리팩토링해줘. auth.ts, user.ts, session.ts, 
  middleware.ts, db.ts 모두 영향받음. 보안 검토도 필요합니다.

🧠 모델 라우팅 v2 추천:
  ▸ 모델: opus (xhigh)
  ▸ Complexity: 10.0/10
  ▸ 예상 비용: $2.93 (입력 40,027토큰, 출력 31,000토큰)
  ▸ 이유: 복잡 키워드 3개 보안 키워드 3개 파일 5개 언급
  ⚠️ 복잡도 7+ — 3단계 분해 권장 (plan→impl→verify), 분해 시 비용 -50% 가능
  💡 대안: sonnet (-80% 비용 변화)

[사용자가 분해 결정 — /spec 으로 계획부터 시작 권장]
```

해석:
- 너무 큰 작업이라 **그냥 한 번에 처리하면 $2.93** 듭니다.
- **3단계로 분해**하면 plan(Opus) → 구현(Sonnet) → 검증(Sonnet) 으로 -50% 절감 가능.
- 또는 **그냥 Sonnet으로 처리하면 -80%** 가능 (품질은 조금 낮음).
- 사용자가 trade-off 보고 결정.

### 예제 4️⃣ — 비용 한도 접근 (경고)

세션 누적 비용이 $4 (한도 $5의 80%) 도달하면:

```
[도구 실행 중 stderr 출력]
⚠️ Budget warning: session 80% ($4.02/$5), daily 23% ($4.61/$20)

[계속 진행됨 — 100% 초과 시에만 차단]
```

세션 누적이 $5 초과하면:

```
[다음 도구 호출 시도 시]
🚨 Budget exceeded: session $5.12/$5, daily $5.71/$20

❌ Tool call blocked. Set CLAUDE_BUDGET_MODE=warn to disable, 
   or raise CLAUDE_BUDGET_SESSION_USD / CLAUDE_BUDGET_DAILY_USD.
```

여기서 사용자는:
- 그대로 멈추기 (이성적 결정)
- 한도 올리기: `export CLAUDE_BUDGET_SESSION_USD=10` 후 재시도
- 임시 해제: `export CLAUDE_BUDGET_MODE=warn`

### 예제 5️⃣ — Tool 반복 감지 (ping-pong)

같은 파일을 5번째 수정하려고 하면:

```
[Edit 도구 실행 후 stderr]
🔄 Tool ping-pong detected: Edit repeated 5 times. Consider rethinking approach.
```

이때 사용자가 클로드에게 **"멈추고 접근 방식 다시 생각해줘"** 라고 지시할 수 있습니다.

### 예제 6️⃣ — 작업 종료 시 LLM Judge

세션이 끝나면 (작업 완료 또는 종료 시점):

```
[Stop hook 실행 중 stderr]
🚨 LLM judge would BLOCK (mode=warn): Tests were weakened with toBeTruthy

   Concerns: 
   - foo.test.ts:42 toBe(value) → toBeTruthy() 변경됨
   - bar.test.ts:18 expect.assert 제거됨
```

해석:
- LLM Judge가 **테스트가 약화됐다**고 감지했습니다.
- `mode=warn` 이라서 차단은 안 하지만 경고 메시지.
- 사용자가 `mode=block` 으로 바꾸면 차단까지 됩니다.

이게 **결정적 verifier(tsc, ruff)만으론 못 잡는 패턴**의 대표 예시.

### 예제 7️⃣ — 캐시 hit rate 알림

세션이 끝났는데 캐시 활용률이 낮으면:

```
[Stop hook 실행 중 stderr]
⚠️ Prompt cache hit rate 35% < 70% — settings.json의 cache_control 확인
```

해석:
- 캐시가 35%만 활용됨. 100% 가까이 활용하면 토큰 80% 절감 가능.
- `settings.json`의 system prompt에 `cache_control: {ttl: "1h"}` 추가 권장.

## ❓ 자주 묻는 질문 (입문자용)

**Q1. 저는 비개발자인데 이걸 써도 되나요?**
A. 네. 클로드 코드를 쓰신다면 모두 해당됩니다. 비용 폭주는 비개발자가 더 위험합니다 — 코드 디버깅 도중 사고가 발생하므로.

**Q2. 적용했다가 안 맞으면 되돌릴 수 있나요?**
A. 네. SETUP_GUIDE 9-7번(rollback)에 백업 복원 방법이 나와 있습니다. 환경변수 한 줄로 전체 비활성화도 가능합니다.

**Q3. 비용이 늘어나지는 않나요?**
A. 오히려 줄어듭니다.
- `prompt-cache-monitor` 후크 자체는 비용 0
- 권장하는 `cache_control: 1h` 설정이 실제 비용을 -67~80% 줄여드림
- `llm-judge`만 미세하게 추가 (세션당 ~$0.001)
- 가장 큰 절감은 `budget-gate`가 사고 방지 (사례: $47,000 → $5)

**Q4. AI 잘 모르는데 "에이전트", "훅", "스킬" 차이가 헷갈려요.**
A. 3줄 요약:
- **훅** = 자동 (운전자가 신경 안 써도 작동)
- **에이전트** = 위임 (운전자가 부르면 옴)
- **스킬** = 매뉴얼 (운전자가 따라야 할 절차)

**Q5. 이게 회사 정책상 외부 API 호출 금지면 어떡하죠?**
A. `CLAUDE_JUDGE_DISABLE=1` 환경변수로 `llm-judge`만 끄시면 됩니다. 나머지 7개 훅은 모두 로컬에서만 동작합니다.

**Q6. 다른 사람과 셋업을 공유하고 싶어요.**
A. 이 레포 자체를 fork 또는 clone하시면 됩니다. settings.json은 개인 자산이므로 제외하시고, hooks/agents/memory-schema만 공유하시면 충분합니다.
