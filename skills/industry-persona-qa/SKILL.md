# Industry Persona QA Skill

## 개요

프로젝트가 속한 산업/사업군을 파악한 뒤, 해당 산업의 다양한 이해관계자 페르소나를 자동 생성하여 각 관점에서 QA를 수행합니다. 단일 관점이 아닌 5~8명의 현실적 페르소나가 각자의 입장에서 피드백을 제공하여, 출시 전에 놓치기 쉬운 문제를 사전에 발견합니다.

---

## Phase 1: Context Gathering (맥락 수집)

### 1-1. 기준일 설정

- 현재 날짜를 감지하되 연도는 항상 **2026**으로 표기한다
- 타임존은 **KST (한국 표준시)** 기준이다
- 보고서 상단에 `기준일: 2026-MM-DD (KST)` 형식으로 명시한다

### 1-2. 산업/사업군 확인

사용자에게 다음과 같이 질문한다:

```
프로젝트가 속한 산업/사업군은 무엇인가요?

예시:
  건설 | 뷰티 | 헬스케어 | 핀테크 | 교육 | 물류
  제조 | F&B | 리테일 | 부동산 | 엔터테인먼트 | 농업
  기타 (직접 입력)
```

사용자가 산업을 알려주면 Phase 2로 이동한다. "기타"를 선택한 경우 구체적 설명을 추가로 요청한다.

### 1-3. QA 대상 확인

사용자에게 QA 대상 유형을 확인한다:

```
QA 대상은 무엇인가요?

  - UI/UX (화면, 흐름, 디자인)
  - API (엔드포인트, 응답 구조)
  - 기획서/PRD (요구사항 문서)
  - 코드 (소스코드 품질)
  - 아키텍처 (시스템 설계)
  - 기타 (직접 입력)
```

### 1-4. 프로젝트 파일 탐색

- 현재 프로젝트의 주요 파일을 읽어 맥락을 파악한다
- README, package.json, 주요 설정 파일, 소스 디렉토리 구조를 확인한다
- QA 대상 파일이 특정되지 않은 경우 사용자에게 경로를 요청한다

## Phase 2: Persona Generation (페르소나 생성)

### 2-1. 페르소나 구성 원칙

산업에 따라 **5~8명**의 페르소나를 생성한다. 각 페르소나는 다음 속성을 가진다:

| 속성 | 설명 | 예시 |
|------|------|------|
| 이름 | 한국식 실명 (성+이름) | 김현장, 박설계 |
| 역할 | 해당 산업의 구체적 직책 | 현장 소장, 설계 엔지니어 |
| 경력 | 신입 / 중급 / 시니어 / 임원 | 시니어 (20년) |
| 핵심 관심사 | 업무에서 가장 중요시하는 것 | 현장 실시간 데이터 |
| 기술 수준 | 낮음 / 보통 / 높음 | 낮음 |
| 사용 환경 | 디바이스, 네트워크, 시간 제약 | 모바일, 4G, 야외 |

### 2-2. 페르소나 다양성 필수 조건

- **경력 분포**: 신입~임원까지 최소 3단계 포함
- **기술 수준 분포**: 낮음~높음까지 최소 2단계 포함
- **엣지 페르소나 1명 이상 필수**: 고령 사용자, 장애가 있는 사용자, 비기술직, 외부 민원인 등 소외되기 쉬운 관점을 반드시 포함
- **외부 이해관계자 1명 이상 필수**: 직접 사용자가 아닌 규제기관, 감사자, 고객, 주민 등

## Phase 3: QA Execution (QA 수행)

### 3-1. QA 항목 체계

QA 대상 유형에 따라 적용할 항목을 선택한다.

#### 공통 QA 항목 (모든 대상에 적용)

| 항목 | 점검 내용 |
|------|----------|
| 접근성 | 이 역할의 사람이 자신의 사용 환경에서 쉽게 사용할 수 있는가? |
| 가치 | 이 역할에게 어떤 구체적 가치를 제공하는가? |
| 위험 | 이 역할 관점에서 어떤 비즈니스/운영 위험이 있는가? |
| 규제 | 해당 산업의 법규/규정을 준수하는가? |
| 경쟁 | 기존에 사용하던 대안(엑셀, 전화, 기존 시스템) 대비 장점이 있는가? |

#### 기술 QA 항목 (코드, API, 아키텍처 대상)

| 항목 | 점검 내용 |
|------|----------|
| 성능 | 이 역할의 전형적 사용 패턴에서 성능 병목이 있는가? (예: 현장 4G에서 대용량 도면 로딩) |
| 보안 | 이 역할이 다루는 데이터가 적절히 보호되는가? (예: 환자 정보 암호화) |
| 확장성 | 이 역할의 사용량이 10배 증가해도 대응 가능한가? |
| 에러 처리 | 이 역할이 에러를 만났을 때 이해 가능한 메시지가 나오는가? |
| 오프라인 | 네트워크 불안정 환경(공장, 현장, 지하)에서 동작하는가? |

#### 기획 QA 항목 (PRD, 기획서, 와이어프레임 대상)

| 항목 | 점검 내용 |
|------|----------|
| 사용자 시나리오 | 이 역할의 실제 일일 워크플로우와 설계된 흐름이 맞는가? |
| 누락 기능 | 이 역할에게 반드시 필요하지만 빠진 기능이 있는가? |
| 우선순위 | 이 역할 관점에서 기능의 우선순위가 적절한가? |
| 용어 | 사용된 용어가 이 역할에게 익숙한가? (전문 용어 vs 일상 용어) |
| 온보딩 | 이 역할이 처음 사용할 때 학습 곡선이 적절한가? |

### 3-2. 페르소나별 QA 수행 절차

각 페르소나에 대해 순서대로 수행한다:

1. **역할 몰입**: 해당 페르소나의 배경, 관심사, 기술 수준, 사용 환경을 기반으로 관점을 전환한다
2. **시나리오 구성**: 이 역할이 실제로 수행할 가장 빈번한 3가지 시나리오를 정의한다
3. **항목별 평가**: 공통 + 유형별 QA 항목을 5점 척도로 평가한다
4. **핵심 피드백**: 가장 중요한 피드백 2~3개를 구체적으로 서술한다
5. **요청 사항**: 이 역할이 반드시 원할 기능 또는 개선 사항을 명시한다
6. **한 줄 소감**: 해당 역할의 말투로 한 줄 소감을 작성한다

### 3-3. 평가 척도

| 별점 | 의미 | 기준 |
|------|------|------|
| 1 | 사용 불가 | 이 역할이 사용할 수 없는 수준 |
| 2 | 불편함 | 사용 가능하나 심각한 불편 |
| 3 | 보통 | 기본 기능은 작동하나 개선 필요 |
| 4 | 양호 | 대부분 만족, 소소한 개선 여지 |
| 5 | 우수 | 이 역할의 기대를 충분히 충족 |

## 설계 원칙

1. **현실성 우선**: 페르소나는 한국 산업 현장의 실제 직책과 역할을 반영한다. 가상의 이상적 사용자가 아닌, 현장에서 만날 수 있는 실존 인물상을 기반으로 한다.
2. **기술 수준 다양성**: 모든 사용자가 기술에 능숙하다고 가정하지 않는다. 특히 현장직, 고령 사용자, 비기술 관리자의 관점을 포함한다.
3. **규제 기반 QA**: 각 산업의 한국 법규와 규제를 반드시 참조한다. 글로벌 기준이 아닌 한국 기준을 우선한다.
4. **한국어 중심**: 페르소나 이름, 피드백, 소감은 모두 한국어로 작성한다. 기술 용어는 업계에서 통용되는 표현을 사용한다.
5. **엣지 케이스 포함**: 장애가 있는 사용자, 고령 사용자, 외부 민원인 등 소외되기 쉬운 관점을 반드시 1명 이상 포함한다.
6. **QA 깊이 매칭**: 코드 대상에는 기술적 QA를, PRD 대상에는 비즈니스 QA를 수행한다. 대상과 맞지 않는 깊이의 QA는 피한다.
7. **구체성**: "UX가 나쁘다" 같은 추상적 피드백이 아닌, "현장 소장이 4G 환경에서 도면을 열 때 30초 이상 로딩되면 다시 전화로 확인하게 된다" 같은 구체적 상황 기반 피드백을 제공한다.
8. **액션 지향**: 보고서의 모든 피드백은 개선 액션으로 전환 가능해야 한다. "문제가 있다"에서 끝나지 않고, "이렇게 개선하면 된다"까지 제시한다.

## 주의 사항

- 페르소나가 기계적으로 동일한 피드백을 내지 않도록 한다. 각 역할의 고유한 관심사와 관점이 반영되어야 한다.
- 기술 수준이 낮은 페르소나의 피드백은 기술 용어 없이 일상 언어로 작성한다.
- 임원급 페르소나는 비용, ROI, 리스크 관점에 집중한다.
- 규제 담당 페르소나는 구체적 법조항이나 규정명을 언급한다.
- 엣지 페르소나는 접근성, 포용성 관점의 피드백을 우선한다.
- 산업 템플릿은 참고용이며, 실제 프로젝트 맥락에 따라 페르소나를 조정한다.
- "기타" 산업 선택 시, 사용자 설명을 기반으로 해당 산업의 이해관계자를 자체 분석하여 페르소나를 생성한다.
