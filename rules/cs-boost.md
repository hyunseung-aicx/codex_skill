# CS & 코딩 역량 강화 규칙

## 1. 코드 변경 시 — 강제 diff 요약 (ALWAYS)

파일을 수정(Edit/Write)할 때마다, 변경 직후 **반드시** 아래 형식으로 1~3줄 요약을 출력한다:

```
📌 변경: [파일:라인] [무엇을] → [어떻게] 바꿈
   WHY: [왜 이렇게 해야 하는지 1줄]
   CS: [관련 CS 개념이 있으면 1줄 — 없으면 생략]
```

예시:
```
📌 변경: mlops.py:398 execute_query → execute_readonly_query
   WHY: 클래스에 execute_query 메서드가 없음 (AttributeError). 실제 메서드명은 execute_readonly_query
   CS: Python은 동적 타입이라 이런 오타가 런타임에야 발견됨 → TypedDict나 Protocol로 방지 가능
```

이건 Insight 블록과 별개. Insight는 새 패턴일 때만, 이건 **모든 변경에 무조건**.
사용자가 "생략" 하면 해당 세션에서만 중단.

## 2. 에러 발생 시 — 3-Layer 분해 (ALWAYS)

에러가 발생하면 바로 고치지 말고, 먼저 이 포맷으로 분해한다:

```
🔍 에러 분해:
   WHAT: [에러 메시지 핵심 1줄]
   WHERE: [어디서 터졌나 — 파일:라인]
   WHY: [근본 원인 — "왜?" 2단계]
   FIX: [수정 방향]
   PREVENT: [재발 방지 1줄]
```

사용자가 에러 메시지를 복붙했을 때, 바로 "고칠게요" 하지 말고 이 분해를 먼저 보여준다.
**사용자가 읽을 시간 3초를 준 후** 수정 진행.

## 3. 커밋 전 — 보안 체크리스트 (ALWAYS)

`git commit` 실행 전에 반드시:

```
🔐 보안 체크:
   [ ] staged 파일에 .env, 비밀번호, 토큰, API 키 없음
   [ ] 하드코딩된 IP/credential 없음
   [ ] SQL 문자열 보간 없음 (파라미터 바인딩 사용)
```

위반 발견 시 커밋을 **중단**하고 경고.

## 4. 주 1회 — CS 약점 리마인더

세션 시작 시 월요일이면, 최근 세션에서 발견된 CS 약점 1개를 상기시킨다:

```
💡 이번 주 CS 포인트: [개념]
   지난주 [상황]에서 이 개념이 관련됐습니다.
   핵심: [2줄 설명]
   실습: [이 프로젝트에서 확인할 수 있는 곳]
```

## 5. diff 읽기 유도

커밋 직전에 전체 diff를 보여주되, **핵심 변경 3개**를 하이라이트하고 퀴즈 형식으로:

```
📝 이번 커밋 핵심 변경 3개:
   1. [파일:라인] — [변경 요약]
   2. [파일:라인] — [변경 요약]
   3. [파일:라인] — [변경 요약]

   Q: 2번에서 왜 [X] 대신 [Y]를 선택했을까요?
   (답을 알면 커밋, 모르면 "왜?" 입력)
```

"왜?" 입력하면 설명. 답을 알면 바로 커밋 진행.
**사용자가 "ㅇㅇ"으로 넘기는 것도 허용** — 강제는 역효과.

- 에러에서 배운 것
- 아키텍처 결정의 트레이드오프

`~/.claude/skills/learned/` 에 저장하여 누적.
